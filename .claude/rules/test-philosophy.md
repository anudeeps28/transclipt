# Testing Philosophy

This rule defines the testing philosophy for all code generated by this harness. Every skill and agent must follow these principles.

---

## Core Principle

**A feature is not done until it is tested.** Not "compiles" — tested. A task marked ✅ means:
1. The code compiles (build passes)
2. Unit tests for the changed code pass
3. The feature works as intended (acceptance criteria met)
4. Existing features still work (regression check)

If any of these fail, the task is not done.

---

## Three Levels of Testing

### Level 1 — Unit Tests (per task)

Every code change must have corresponding unit tests. These run as part of the task's `<verify>` command.

- Test the public interface of the changed code
- Test happy path + key edge cases
- Mock only external boundaries (DB, HTTP, file I/O) — never mock internal classes
- If a task changes a method, there must be a test that exercises that method

**Gate:** A task's `<verify>` command includes running relevant unit tests. The task is only ✅ when both build and unit tests pass.

### Level 2 — Integration Tests (per feature)

After all tasks in a story/feature are complete, integration tests verify the components work together.

- Test the feature end-to-end through the actual system (API → service → database, or UI → API → service)
- Use real dependencies where practical (test databases, emulators, in-memory providers)
- Test that the feature interacts correctly with existing components
- Test that existing features are not broken by the new code

**Gate:** Integration tests must pass before the feature can proceed to evaluation/PR.

### Level 3 — Acceptance Tests (per feature)

Verify what the user actually sees and experiences. These are derived from the acceptance criteria defined in the test strategy.

- Each acceptance criterion from the plan becomes a testable scenario
- Test from the user's perspective: "User does X → sees Y"
- Can be automated (API calls, UI automation) or structured manual verification
- Must cover the golden path and critical edge cases

**Gate:** Acceptance criteria must be verified (PASS/FAIL per criterion) before PR.

---

## Test Strategy Is a Planning Artifact

Every plan must include a test strategy. This is not optional. The test strategy defines:

1. **Acceptance criteria** — What proves the feature works, written as testable scenarios
2. **Integration test scenarios** — How the feature interacts with other components
3. **Regression guardrails** — What existing behavior must NOT change
4. **Test tasks** — Explicit tasks in the plan for writing tests

The test strategy is produced during planning (Phase 2) and verified during evaluation (Phase 3.6).

---

## Test Tasks Are Mandatory

Every plan must include explicit test-writing tasks (`type="test"`). These are real tasks in the plan, not afterthoughts:

- At minimum: unit tests for every new/changed public method
- For features touching multiple components: integration tests
- For user-facing features: acceptance test scenarios

Test tasks should be in the same or next wave after the code they test — never deferred to "we'll write tests later."

---

## Verify Commands Include Tests

The `<verify>` field on every task must include running relevant tests, not just building:

- **Wrong:** `dotnet build src/Project.csproj`
- **Right:** `dotnet build src/Project.csproj && dotnet test src/Project.Tests.csproj --filter "RelevantClass"`

The plan agent generates verify commands that include test execution. The specific test command syntax comes from `tasks/lessons.md`.

---

## Regression Is Everyone's Job

Every change must be checked for regression:

- Unit tests catch method-level regression
- Integration tests catch component-interaction regression
- The evaluator checks for unintended side effects in the diff
- The acceptance-test-agent verifies existing behavior is preserved

If a test breaks and it's not related to the current feature — that's a regression. Fix it before proceeding, don't skip it.

---

## What "Stack-Agnostic" Means

This philosophy applies to any language, framework, or stack. The specific test commands, frameworks, and conventions are defined in `tasks/lessons.md` — not hardcoded anywhere:

- .NET → xUnit/NUnit, `dotnet test`
- Node.js → Jest/Vitest, `npm test`
- Python → pytest, `python -m pytest`
- Go → `go test ./...`
- Angular/React → Cypress/Playwright for e2e, Jest for unit
- Any other stack → define your commands in `tasks/lessons.md`

The harness reads commands from `lessons.md` and executes them. The principles are universal.
